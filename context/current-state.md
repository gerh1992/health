# Current state

_Last updated: 2026-06-09_

## Main objective
Optimizar performance atlética, especialmente:
- velocidad,
- agilidad,
- aceleración,
- fuerza.

No optimizar el sistema como si fuera un proyecto de estética corporal.

## Current active subsystem
Hoy el subsistema más desarrollado es `performance/`, que registra:
- sueño,
- biometría simple de recovery,
- entrenamiento,
- suplementos,
- resúmenes semanales.

## What is already tracked
- Horas dormidas
- Calidad subjetiva de sueño
- Energía subjetiva
- HRV
- RHR
- sesiones de gym/pádel/recovery
- detalles KPI relevantes del entrenamiento cuando existen
- adherencia a suplementos reportados

## Interpretation stance
Los datos diarios no deben interpretarse aislados.
La lectura correcta usa:
- baseline personal evolutivo,
- desvío de hoy,
- densidad de carga reciente,
- consistencia de sueño,
- percepción subjetiva,
- y contexto de entrenamiento.

## Current recurring accountability cadence
- 09:00 local: sleep/recovery check-in
- 22:00 local: training + creatine/protein/collagen check-in
- 23:30 local: magnesium/ashwagandha/retinol check-in
- weekly summary: adherence/performance review

## Known current limitations
- El repo todavía no tiene una capa médica completa.
- No hay todavía un modelo formal para labs, diagnósticos, lesiones, medicación o antecedentes clínicos.
- No todo el contexto de salud importante está cargado todavía.
- Hay que distinguir entre “no documentado” y “negativo/no existe”.

## Working rule for future additions
Agregar información solo si mejora decisiones.
No meter datos “por completitud” si después nadie los usa.

## Good next additions
Cuando estén disponibles, sumar en `medical/` o `context/`:
- antecedentes relevantes,
- lesiones importantes,
- medicación/suplementación crónica,
- estudios de laboratorio,
- imágenes/informes,
- umbrales o flags relevantes para toma de decisiones.
